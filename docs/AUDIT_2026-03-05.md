## Detailed Findings

### [HIGH] #1: Health Check Endpoint Missing Rate Limiting

**Location:** `src/app/api/health/route.ts:22`
**Verified:** Yes — confirmed no rate limiting import or check in file
**Risk:** `/api/health` makes 2 external requests (Snowtrace + Routescan) per call without rate limiting. An attacker can spam this endpoint to exhaust upstream API rate limits, degrading service for legitimate users.
**Impact:** Denial of service via reflected traffic amplification
**Fix:**
```typescript
import { isRateLimited } from '@/lib/rate-limit';

export async function GET(request: NextRequest) {
  const ip = request.headers.get('x-forwarded-for')?.split(',')[0]?.trim() || 'unknown';
  if (isRateLimited(ip)) {
    return NextResponse.json({ status: 'rate_limited' }, { status: 429 });
  }
  // ... existing logic
}
```
**Effort:** S
**Status:** FIXED

### [HIGH] #2: Cache Pollution via Unvalidated Period in Cache Key

**Location:** `src/app/api/contract/[address]/analytics/route.ts:39-42`
**Verified:** Yes — `period` param is not validated before use in cache key
**Risk:** The `period` parameter accepts any string value. While `periodDays` defaults to 30 for invalid values, the cache key uses the raw string: `analytics:${address}:${period}`. An attacker sending requests with random period values (`?period=abc`, `?period=xyz123`, etc.) creates unique cache entries, filling the 500-entry cache with garbage and evicting legitimate entries.
**Impact:** Cache DoS — legitimate cached analytics get evicted, forcing expensive recomputation
**Fix:**
```typescript
const periodParam = searchParams.get('period') || '30d';
if (!['7d', '30d', '90d'].includes(periodParam)) {
  return NextResponse.json(
    { success: false, error: 'Invalid period. Use 7d, 30d, or 90d.' },
    { status: 400 }
  );
}
```
**Effort:** S
**Status:** FIXED

### [MEDIUM] #3: Missing Content-Security-Policy Header

**Location:** `next.config.ts:4-13`
**Verified:** Yes — only X-Content-Type-Options, X-Frame-Options, Referrer-Policy are set
**Risk:** Without CSP, if a dependency (recharts, viem) has an XSS vulnerability, there's no defense-in-depth to restrict script execution.
**Impact:** Increased XSS blast radius if a dependency is compromised
**Fix:** Add CSP header to next.config.ts:
```typescript
{
  key: 'Content-Security-Policy',
  value: "default-src 'self'; script-src 'self' 'unsafe-inline' 'unsafe-eval'; style-src 'self' 'unsafe-inline'; img-src 'self' data:; connect-src 'self' https://api.snowtrace.io https://api.routescan.io; font-src 'self';"
}
```
Note: `unsafe-inline` and `unsafe-eval` are needed for Next.js. Use nonce-based CSP for stricter policy if possible.
**Effort:** S
**Status:** FIXED

### [MEDIUM] #5: Cache Eviction Is FIFO, Not LRU

**Location:** `src/lib/cache.ts:22-24`
**Verified:** Yes — `cache.keys().next().value` gets insertion-order first key
**Risk:** When cache is full, the oldest-inserted entry is evicted regardless of access recency. Frequently accessed entries can be evicted if they were inserted early, while rarely-accessed recent entries survive.
**Impact:** Suboptimal cache hit rate under load. Not a security issue per se, but reduces performance under cache pressure.
**Fix:** Update entry on read to move it to end of Map (makes it truly LRU):
```typescript
export function getCached<T>(key: string): T | null {
  const entry = cache.get(key);
  if (!entry) return null;
  if (Date.now() > entry.expiry) {
    cache.delete(key);
    return null;
  }
  // Move to end for LRU behavior
  cache.delete(key);
  cache.set(key, entry);
  return entry.data as T;
}
```
**Effort:** S
**Status:** FIXED

### [MEDIUM] #6: ABI JSON.parse Without Array Validation

**Location:** `src/lib/apis/snowtrace.ts:42`
**Verified:** Yes — `JSON.parse(json.result)` with no type check on result
**Risk:** If Snowtrace returns malformed or non-array ABI data, `JSON.parse` succeeds but downstream code (viem's `decodeFunctionData`) receives unexpected input. The `try/catch` in functions.ts:29 mitigates crash risk, but parsing a huge non-array object could be expensive.
**Impact:** Low — viem's try/catch handles gracefully, but adding validation is defense-in-depth
**Fix:**
```typescript
const abi = JSON.parse(json.result);
if (!Array.isArray(abi)) return null;
```
**Effort:** S
**Status:** FIXED

### [MEDIUM] #7: Infinity Stored in Function Stats

**Location:** `src/lib/processing/functions.ts:40`
**Verified:** Yes — `minGas: Infinity` is the initial value, converted to 0 at line 62 only during output mapping
**Risk:** The in-memory Map stores `Infinity` as a value. `JSON.stringify(Infinity)` returns `null`. If cached analytics are serialized and deserialized, `minGas` becomes `null` instead of `Infinity`, breaking `< Infinity` comparison on next use. Currently not an issue because cache stores the final mapped output, but fragile.
**Impact:** Low — currently handled correctly at output, but could break if cache structure changes
**Fix:** Use `Number.MAX_SAFE_INTEGER` instead of `Infinity`:
```typescript
minGas: Number.MAX_SAFE_INTEGER,
```
**Effort:** S
**Status:** FIXED

### [LOW] #8: parseInt Without Radix Parameter

**Location:** Multiple files — `analytics.ts:15,27,56`, `functions.ts:34`, `callers.ts:24-25`, `errors.ts:15`
**Verified:** Yes — all `parseInt()` calls lack explicit radix
**Risk:** `parseInt` without radix defaults to 10 for decimal strings, but strings starting with `0x` are parsed as hex. Transaction data from Routescan uses decimal strings, so this works correctly in practice. However, explicit `parseInt(value, 10)` is safer.
**Impact:** Negligible — Routescan always returns decimal strings
**Fix:** Add radix 10 to all parseInt calls:
```typescript
parseInt(tx.timeStamp, 10)
parseInt(tx.gasUsed || '0', 10)
```
**Effort:** S
**Status:** FIXED

### False Positives Rejected from Agent Reports:

1. **"CRITICAL: IP Spoofing via x-forwarded-for"** — Downgraded to LOW. On Vercel (the deployment target), x-forwarded-for is set by the edge network and cannot be spoofed by clients. This would be critical for self-hosted deployments only.

2. **"CRITICAL: Unbounded Array Processing DoS"** — Rejected. Routescan API returns max 10,000 transactions per request (hardcoded offset limit). The processing is O(n) with n capped at 10,000 — acceptable for a single request. Rate limiting (30 req/min) prevents abuse.

3. **"HIGH: Integer Overflow in Gas Calculations"** — Rejected. JavaScript's Number.MAX_SAFE_INTEGER is 9,007,199,254,740,991. Even 10,000 transactions at 30M gas each = 300B, well within safe range. No overflow risk.

4. **"HIGH: Sort Parameter Not Whitelisted"** — Rejected. Code at transactions/route.ts:30 uses strict ternary: `=== 'asc' ? 'asc' : 'desc'`. Only these two strings ever reach the API URL. This is already a whitelist.

5. **"HIGH: Page/Offset Parameter Injection"** — Downgraded/Rejected. Math.max/Math.min bounds checking is present. Values are integers passed as query params to Routescan's standard Etherscan-compatible API. No injection vector exists.

6. **"MEDIUM: Cache Timing Attack"** — Rejected. Cache/non-cache timing difference reveals only whether a public contract was recently queried. All contract data is public on-chain. No sensitive information is exposed.

7. **"MEDIUM: Missing viem Decode Timeout"** — Rejected. viem's decodeFunctionData is a synchronous pure function that parses hex data against ABI — it completes in microseconds. No timeout needed.

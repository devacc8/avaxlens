## Detailed Findings

### [HIGH] #1: Unbounded Cache Growth (Memory Leak)

**Location:** `src/lib/cache.ts:1-15`
**Verified:** Yes — Read confirmed no size limits, no periodic cleanup
**Risk:** In-memory Map grows indefinitely. Each unique address creates multiple cache entries (txs, abi, source, info, analytics per period). Expired entries are only cleaned on read access. An attacker can scan thousands of addresses to exhaust server memory.
**Impact:** Server crash / Vercel function memory limit exceeded
**Fix:**
```typescript
const MAX_CACHE_SIZE = 500;

export function setCache(key: string, data: unknown, ttlMs: number): void {
  if (cache.size >= MAX_CACHE_SIZE) {
    // Evict oldest entry
    const firstKey = cache.keys().next().value;
    if (firstKey) cache.delete(firstKey);
  }
  cache.set(key, { data, expiry: Date.now() + ttlMs });
}
```
**Effort:** S
**Status:** FIXED in v0.4.1 — MAX_CACHE_SIZE=500, expired cleanup, LRU eviction

### [HIGH] #2: No Rate Limiting on API Routes

**Location:** `src/app/api/contract/[address]/analytics/route.ts`, `info/route.ts`, `transactions/route.ts`
**Verified:** Yes — no rate limiting middleware exists in any route
**Risk:** External APIs (Snowtrace, Routescan) have 2 req/sec and 10K/day limits. An attacker can make unlimited requests to your API routes, exhausting the external API quota and degrading service for all users.
**Impact:** Service unavailability due to exhausted API quotas
**Fix:** Add simple in-memory rate limiter or use Vercel's built-in rate limiting. For hackathon, a simple IP-based counter with sliding window would suffice.
**Effort:** M
**Status:** FIXED in v0.4.1 — sliding window rate limiter (30 req/min per IP), added to all 3 API routes

### [HIGH] #3: Missing Security Headers

**Location:** `next.config.ts:3`
**Verified:** Yes — config is empty: `const nextConfig: NextConfig = {};`
**Risk:** Missing X-Frame-Options (clickjacking), X-Content-Type-Options (MIME sniffing), Referrer-Policy, CSP
**Impact:** Standard web security vulnerabilities
**Fix:**
```typescript
const nextConfig: NextConfig = {
  headers: async () => [
    {
      source: '/:path*',
      headers: [
        { key: 'X-Content-Type-Options', value: 'nosniff' },
        { key: 'X-Frame-Options', value: 'DENY' },
        { key: 'Referrer-Policy', value: 'strict-origin-when-cross-origin' },
      ],
    },
  ],
};
```
**Effort:** S
**Status:** FIXED in v0.4.1 — X-Content-Type-Options, X-Frame-Options, Referrer-Policy added

### [MEDIUM] #4: Missing Query Parameter Validation in Transactions Route

**Location:** `src/app/api/contract/[address]/transactions/route.ts:19-21`
**Verified:** Yes — `parseInt` without validation, `sort` without whitelist
**Risk:** `parseInt('abc')` returns `NaN`, negative values pass through, `sort` param is unvalidated. These values are passed to Routescan API URL construction.
**Impact:** Malformed external API requests, unexpected behavior
**Fix:**
```typescript
const page = Math.max(1, parseInt(searchParams.get('page') || '1') || 1);
const offset = Math.min(10000, Math.max(1, parseInt(searchParams.get('offset') || '10000') || 10000));
const validSorts = ['asc', 'desc'];
const sort = validSorts.includes(searchParams.get('sort') || '') ? searchParams.get('sort')! : 'desc';
```
**Effort:** S
**Status:** FIXED in v0.4.1 — Math.max/min bounds, sort whitelist

### [MEDIUM] #5: Missing Response Schema Validation in TransactionsTab

**Location:** `src/components/dashboard/TransactionsTab.tsx:35-38`

**Risk:** If API returns unexpected structure, `processRawTransactions()` crashes with unhandled error

```typescript
if (json.success && Array.isArray(json.data?.transactions)) {

  setTransactions(processed);
} else {
  setError('Failed to load transactions');
}
```
**Effort:** S
**Status:** FIXED in v0.4.1 — Array.isArray guard added

### [MEDIUM] #6: No Client-Side Fetch Timeout

**Location:** `src/components/dashboard/TabNavigation.tsx:38`, `src/components/dashboard/TransactionsTab.tsx:30`
**Verified:** Yes — client-side fetch calls use native `fetch()` without timeout, while server-side uses `fetchWithTimeout()`
**Risk:** If backend hangs, user sees indefinite loading spinner
**Impact:** Poor UX; user stuck on loading state
**Fix:** Use `AbortController` with 15s timeout on client-side fetch calls
**Effort:** S
**Status:** FIXED in v0.4.1 — AbortController with 15s timeout on all client fetches
